★ What's running right now · Live Daemon

This is what the dashboard you're looking at is reading from. Everything below is built, deployed, and running 24/7 on the alice Oracle ARM VM (ubuntu@129.153.22.182). The rest of this document (sections 0–16) is the original data-source catalog and build-order plan; this section is the operational reality.

Architecture

A single Python asyncio process — live_daemon.py — runs forever under systemd. It owns three concurrent loops:

  • Scheduler. 11 cron-style periodic scans, each invoked by loop.run_in_executor(None, mod.run) on its own cadence. Cron timing mirrors the dashboard's SCAN_REGISTRY.
  • Daemons. Long-running coroutines that persist a connection (currently just rvol_stream, the Alpaca IEX websocket). Each gets a stop_event: asyncio.Event for cooperative shutdown and is re-spawned on crash.
  • HTTP server. Serves the dashboard at http://<host>:8420/, exposes /api/state (full bundle), /api/stream (Server-Sent Events: a scan event on each scan completion plus data events on any data/live/*.json mtime change), and POST /api/run/{scan} for manual triggers.

Events from any scan land in a shared in-memory bus, get appended to data/live/events.json, and stream out over SSE in real time. Each scan also writes its own sidecar JSONs under data/live/ (watchlist, hits, status, plus scan-specific files). Restart latency is ~100–300ms thanks to a shared shutdown_event that kicks SSE clients out of their 15s keep-alive immediately.

The 12 scans

How to read "top-N watchlist" in the Universe column. Two scans still examine only the top-N tickers from watchlist.json (itself a derived ranking of every ticker that has scored points from any other scan, decayed with a 12-hour half-life): options_unusual (top-60, yfinance ~2 s/ticker) and the rvol_stream daemon (top-25, Alpaca IEX websocket capped at 30 symbols). For these two, if a stock has never scored from any upstream scan, it will never receive an unusual-options check or an intraday-RVOL subscription — a deliberate closed loop that respects rate limits and connection caps. The Fundamentals scan was moved upstream on 2026-05-28 and now slices the full universe; broad-universe daily RVOL is now detected inside tech_slice alongside its setup patterns.

Reading a scan tile

Each tile in the scan strip shows three lines: the friendly title, the cadence, and a counter line that looks like 4m ago · 83ev · 72aw. The breakdown:

  • 4m ago — wall-clock age of the most recent completed run for this scan. Switches to now for runs within the last minute, hours for older runs, days if the scan hasn't fired (which usually means a stale daemon — the tile will turn yellow).
  • 83ev — number of events emitted during that run. An event is any line the scan wrote to the live feed (one per ticker processed, one per filing examined, etc.). High ev just means the scan had a lot to look at, not that anything interesting happened. For tech_slice this will be ~400 every run (one per ticker in the slice); for halt_tape on a quiet day it might be 0.
  • 72aw — number of awards from that run. An award is when a ticker actually scored points — i.e. it matched one of the scan's setup conditions. aw ≤ ev always. The award/event ratio tells you the scan's noise floor: 0/400 is silence, 72/83 means almost everything triggered (which is what we want to lower for xbrl_facts's current 68% award rate — see the improvements section below).

Tile colors: scan-specific accent = healthy, ran recently. Pulsing dot in the corner = ran within the last 90 seconds. Yellow border = stale (overdue based on the scan's expected cadence). Red background = the last run errored — hover the tile to see the exception message. Muted/grey = pending, never run yet on this process.

Dashboard (index.html)

  • Top bar shows the live regime label (e.g. “REGIME · CONSTRUCTIVE · ×1.1”) and a UTC clock. Color matches the regime (green/yellow/muted/red).
  • Scan strip — a 2×6 grid of health tiles. Each tile shows the friendly title, cadence, last-run age, and events/ awards counts. Left border = scan-specific accent color when healthy, yellow when stale, red on error. The pulsing dot in the corner indicates a run within the last 90 seconds.
  • Scan feed (left column) streams every event as it arrives via SSE. Rows are color-coded by source scan (border + text tint), with notable / hit events overriding the per-scan tint in accent yellow / hit pink. Hover a row to see the scan id. On page refresh the entire visible backlog is rendered instantly and the view snaps to the latest event — no scroll-from-top animation. Live events are then paced through a queue that aims to spread them across a ~15-second drain window (min 120 ms, max 2.5 s between rows), so a 60-event burst from a finishing scan feels like a steady ticker instead of a wall of text.
  • Top-10 watchlist (right column) is a time-decayed aggregate of recent awards. Each row shows rank, ticker, score, a 24-point sparkline (green if trending up, red if down), and a horizontally-scrolling marquee of attribute tags below — one chip per distinct attribute that has scored the ticker recently (e.g. triple_8k, activist_13d, revenue_acceleration). Tag colour matches the source scan; hover any row to pause the marquee.
  • Scores decay with a 12-hour half-life. Every award adds points; every poll re-applies the decay score × 0.5(elapsed / 12h). So a single +20 award is worth +20 immediately, +10 after 12 hours, +5 after a day, and is auto-pruned below 0.1. Tickers slip off the watchlist organically as their signals age out — there is no manual eviction.
  • Attribute tags decay faster than the score itself (24-hour half-life, pruned when normalized weight < 0.15) so a ticker's tag set reflects what's currently driving it, not what happened two days ago.
  • Hit banner + chime — any award labeled "hit" slides a banner across the top and plays a two-tone WebAudio chime (gated on first user click per browser autoplay policy).
  • Regime-change banner — when macro_regime flips, a center-top banner shows the transition (e.g. "NEUTRAL → CONSTRUCTIVE ×1.1") for 6 seconds and triggers the chime.
  • Footer shows total scans run, total hits today, current regime + VIX level, last scan name + age, daemon status, and links to the repo and this page.

Operational notes

  • Box: alice (Oracle Cloud A1.Flex ARM, 129.153.22.182), free-tier.
  • Systemd: raymond.service, auto-restart with Restart=always / RestartSec=5. EnvironmentFile loads ~/raymond/.env (chmod 600; holds Gemini, Finnhub, Alpaca, FRED keys; not in git).
  • Code sync: alice cron pulls git pull --ff-only every 30 min as user ubuntu. Push to main + manual sudo systemctl restart raymond is the deploy loop. data/live/ is gitignored so the daemon can write freely without race-conditioning the pull.
  • SEC compliance: all EDGAR / data.sec.gov requests send a UA header identifying the project and email; rate-limited to ≤10 req/s with a polite 0.12s gap between fetches inside any scan. insider_cluster has a per-run budget of 600 form fetches.
  • What's been retired: the old .github/workflows/live-scans.yml GHA crons are all commented out (kept as workflow_dispatch fallbacks). The daemon owns the cadence now.
Q1: Alpaca Market Data vs yfinance for technical setups?

Use both, for different jobs. yfinance is the right tool for the nightly daily-bar scan (1y of OHLCV × ~6000 tickers in batched downloads, free, no quota, indicator computation done locally with pandas in tech_scan.py). Yahoo's daily data is split/dividend-adjusted and consensus-quality for the close-of-day swing setups we care about.

Alpaca becomes essential the moment we want intraday or real-time: premarket gappers, intraday RVOL during the trading day, level-1 quotes, minute bars for entry timing, and eventually live execution. We already have Alpaca keys for the news API, so adding bars/quotes is just additional endpoints.

  • yfinance wins: nightly batch daily bars, free fundamentals/holders/earnings, no rate limit grief on bulk downloads, ~6000 tickers in a few minutes.
  • Alpaca wins: intraday minute bars, premarket / after-hours bars, real-time trades & quotes (IEX feed free, SIP $99/mo), bulk multi-symbol bar endpoint, institutional data lineage.
  • Big caveat on free Alpaca: the free tier is IEX-only, which is ~2–3% of consolidated volume. Fine for last-trade snapshots and price discovery on liquid names; bad for accurate RVOL on microcaps. SIP ($99/mo personal) gives the full consolidated tape and is what we'd want if intraday RVOL becomes part of the strategy.
  • Polygon is the third option — $29–$199/mo, better historical depth than Alpaca, includes 13F & financials bundled. Not worth it while yfinance daily + Alpaca news covers us.
Q2: Volume — what did you propose, and what do we actually have?

Already wired into tech_scan.py:

  • vol_ratio_5_60 — last-5d average volume / prior-55d average. > 1.5 is "stepped up." Used in the breakout-day-1 regime classifier and consumed by the BUP++ regime score.

What's still missing on volume that would meaningfully improve signal:

  • Dollar volume floor (close × volume, 20d avg) — gate liquidity properly. A $0.50 ticker doing 5M shares is $2.5M/day — fine for a $500 paper position, dangerous for size. Trivial to add to tech_scan, just one extra field.
  • Float turnover ratio (today's volume / free float) — the canonical squeeze precursor. > 100% turnover in a day is a huge tell. Needs reliable float (Finnhub, not yfinance, for microcaps).
  • OBV / accumulation-distribution slope — directional volume divergence. Compute locally from existing bars, no new data fetch.
  • Up-day vs down-day volume ratio (20d) — cleaner accumulation read than OBV.
  • Intraday RVOL (volume by hour vs typical volume by hour, sampled mid-session) — requires Alpaca minute bars. This is the single biggest add for catching breakouts during the day rather than at close.
  • Premarket / extended-hours volume — only available through Alpaca (yfinance does not surface premarket bars reliably). Big tell on overnight news reactions.

0.5 · Frequency tiers & cascading sieve

My earlier 30-minute estimate was wrong. Real nightly scans take hours, and forcing every dimension onto the same cron is both expensive and wasteful — float doesn't fluctuate hourly, but RVOL does. The right model is multiple crons at different frequencies, where slower tiers sieve the universe that faster tiers scan.

The sieve cutoffs are now driven by information freshness, not by computational shortcuts. A name excluded from intraday scanning because its weekly structural score is poor is genuinely not a candidate today — we're not hiding information, we're matching scan cadence to data freshness.

The cascading sieve

Each tier produces a quality score for its slice of attributes, and emits the universe of names that pass into the next-faster tier. No information is discarded — every ticker still has a row at every tier; the cascade just decides which rows get the expensive faster scans. 

T4 (monthly):    all 6000           → "structurally sound"   universe ~3500
                                       (drops: shells, no fundamentals,
                                        deeply distressed, no float data)
T3 (weekly):     3500                → "tradable swing candidates" ~1500
                                       (drops: no SI/float, illiquid by
                                        dollar volume, no recent options OI)
T2 (daily):      1500                → "active watchlist"      ~300
                                       (drops: bad regime, blown-up,
                                        no recent volume or news, dead chart)
T1 (intraday):   300                 → "today's setups"         ~30–80
                                       (drops: no RVOL pickup, no premarket
                                        interest, sector down hard)
T0 (event):      ANY ticker          → can promote into T1 on a
                                       fresh 8-K, halt resume, or news hit

T0 is a special case — it never drops anyone, it only promotes. A microcap that failed T2's regime filter yesterday but files an 8-K announcing FDA approval at 9:47am jumps straight onto the intraday watchlist for the day. The slow-tier scores are a baseline, not a permanent verdict.

Why this is better than "scan everything nightly"

  • Slow data doesn't waste fast cycles. Float and 13F don't change overnight; pulling them daily on 6000 tickers burns Finnhub quota for no signal gain.
  • Fast data isn't bottlenecked by slow scans. Intraday RVOL on a 300-ticker watchlist runs in seconds. Forcing it through a 6000-ticker pass would add minutes of latency where it matters most.
  • Sieving is justified at every step. A ticker excluded from T1 because it failed T2's regime gate yesterday is not a candidate today; the sieve isn't hiding signal, it's matching scan cadence to decision cadence.
  • Costs are bounded per tier. Each cron has a clear budget — T1 ~3 min, T2 ~30 min, T3 ~3 hours, T4 ~6 hours, all running on raymond on independent schedules.
  • Event-driven promotion preserves serendipity. The T0 poll can promote any ticker into the watchlist regardless of tier history — so a sleeper that wakes up on news isn't trapped by yesterday's structural verdict.
Net result: the BUP++ score we already built becomes the T2 daily score. We add a T3 weekly setup_score that gates which 1500 names T2 even looks at, and a T1 15-min intraday_score that gates which 300 names get live-monitored. The pipeline that exists today is one tier of five — the most important one for the swing-trading decision, but not the only one.

17 · Potential improvements (next on the bench)

Ordered by approximate value-per-hour-of-work. Items marked P1 are quick wins that build on what's already running; P2 are meaningful additions that take more engineering; P3 are deferred / decision-needed.

P1 — quick wins on existing scans

  • xbrl_facts noise floor. The first live run awarded 26 of 38 fetched filings (68% award rate) because the “fresh XBRL refresh” fallback fires whenever there's no prior-year comparable. Either raise the floor (require prior-year data, drop the fallback) or downgrade it to an info-only event with 0 points.
  • xbrl_facts → cash & runway. Same companyfacts JSON also contains CashAndCashEquivalentsAtCarryingValue and NetCashProvidedByUsedInOperatingActivities. Compute cash burn-rate → runway-in-quarters per ticker. Single biggest dilution risk signal we don't yet score.
  • insider_cluster — add dollar-weighted scoring. Right now we count buys vs sells. A $5M CEO open-market buy is very different from three $50K director buys. insider_xml already returns buy_dollars; just weight the award.
  • halt_tape — dedupe consecutive resumes. The feed re-publishes the same resume record across consecutive polls; the scan currently re-awards on each. A persisted set of seen halt-IDs would cut the dashboard noise by ~50%.
  • rvol_stream — carry the prior-day-close in the baseline. The IEX-adjusted baseline drops to zero on tickers that haven't printed an IEX bar yet today, producing infinite RVOL on the first bar. Cap RVOL at 20× or require N≥3 buckets before awarding.
  • Dashboard — click a feed row to pin it. Already-rendered rows scroll out of view as new events arrive. A click-to-pin mechanism + a “pinned” strip above the feed would let you keep a hit visible while you investigate.
  • Dashboard — per-scan mute toggles. The 12 scan tiles are already individually identifiable; one click should hide that scan's events from the feed (e.g. mute halt_tape after-hours when it's noise).

P2 — new scans & signals

  • FINRA short-volume daily (T2). Public FINRA file, one CSV/day, all tickers. Off-exchange short pressure is currently blind — Reg SHO threshold list only catches the worst persistent cases.
  • Float / SI upgrade to Finnhub. yfinance's float is unreliable on microcaps. A fundamentals_snap swap to Finnhub's /stock/profile2 + /stock/social-sentiment would tighten the squeeze-setup signal materially.
  • Analyst revision trend. Finnhub free tier exposes /stock/recommendation and /stock/price-target. A weekly delta scan would catch coverage initiations + PT raises before they hit price.
  • Lockup expiration calendar. Parse the lock-up sections of recent S-1 / S-1/A prospectuses (regex on “Lock-up” section + 180-day default). Surface tickers within 14 days of expiry as catalyst-calendar entries.
  • Premarket gappers daemon. Alpaca IEX delivers premarket bars 4am–9:30am ET. A second daemon (or extension of rvol_stream) that watches for >5% premarket moves on the full top-300 watchlist, not just top-25.
  • Reddit / StockTwits sentiment scan. Both free. Mention-count delta is a better retail-attention proxy than X. Drives the “social” half of catalyst classification.
  • News scan via Alpaca / Benzinga. We have the Alpaca news endpoint keys; not currently wired into the live daemon. Headlines for watchlist tickers should stream as events alongside the structural signals.

P3 — bigger builds / decisions pending

  • 13F holdings (deferred Phase D part 2). Needs a CUSIP→ticker resolution path — OpenFIGI free tier (≤25 req/min, no key needed) is the standard answer. Build is ~3–4 hrs once that decision is made: parse 13F-HR INFORMATION TABLE XML, count new holdings per CUSIP, persist quarterly state, award tickers showing up in many new institutional positions.
  • Catalyst-scoring loop on live events. The legacy market-scan pipeline already uses Gemini to score catalysts on yesterday's news. Routing live hit events through the same scorer in real time (gated to ≤5 hits/hour to control cost) would produce a high-confidence “top catalyst right now” list.
  • Per-scan award decay tuning. Watchlist scores currently decay at a single half-life (~12 hours). Different signals deserve different decays — a halt resume is stale in 30 minutes; an XBRL revenue acceleration is fresh for days.
  • Backtest harness. Every award has a ticker, timestamp, and points. With 30 days of events.json history we could measure forward-return distribution per scan and re-weight scoring empirically instead of by intuition.
  • Alpaca SIP feed (— paid, $99/mo). Only worth it if intraday execution becomes part of the strategy. IEX free is fine for current signal-only use on liquid names; microcap RVOL is structurally noisy until then.
  • Mobile dashboard. Current layout is desktop-first (terminal column needs width). A condensed mobile view (scan strip + watchlist only, feed behind a toggle) would let you glance at the state from a phone.

Maintenance / hygiene

  • Persist events.json rotation. Currently grows unbounded. Rotate at ~10k events or daily; archive older to data/live/archive/events-YYYYMMDD.json.gz.
  • Telemetry on alice. No external alerting if the daemon dies and systemd can't restart it. A 1-line cron on a second host that curls /healthz and pings Pushover on failure would close the loop.
  • Secret rotation. All keys live in ~/raymond/.env on alice. Rotation is manual; quarterly rotation reminder.

What's missing the most

  1. Macro context. We're scoring single names in a vacuum. A 0.7×–1.3× multiplier driven by SPY trend + VIX + credit spreads + 2s10s would change the BUP++ top-of-list meaningfully.
  2. Richer volume features. One ratio (5d/55d) is the floor, not the ceiling. Dollar volume, float turnover, up/down ratio, OBV — all free, all from existing data.
  3. Options-derived signals. No P/C, no IV rank, no OI change. Single biggest under-instrumented dimension.
  4. Macro & sector tailwind. No relative-strength check — we'd take a P2 setup in a sector that's down 8% on the week the same as one in the leading sector.
  5. FINRA short volume. Free, daily, off-exchange. No reason not to ingest.
